# 儿童安全政策

**Signet协议 — v0.1.0**

*模板 — 在部署前，请就您所在司法管辖区寻求具有资质的法律顾问的意见。*

## 1. 目的

本儿童安全政策（"政策"）阐明Signet协议（"我们"或"本协议"）保护与Signet协议（"协议"）互动或受其影响的儿童的承诺。协议的第4级验证（专业验证 — 成人+儿童）专门涉及儿童身份验证，本政策规范该功能及所有与儿童相关数据的处理方式。

儿童的安全和隐私是我们的首要关切。本政策旨在遵守全球所有适用的儿童保护法律，并在切实可行的情况下超越最低要求。

---

## 2. 适用范围

本政策适用于：

- Signet协议中涉及18岁以下（或相关司法管辖区法定成年年龄，以较高者为准）个人的所有互动
- 第4级凭证的签发和验证流程
- 使用零知识证明的年龄验证流程
- 通过协议与儿童数据互动的所有人员、验证方、依赖方和第三方
- 使用协议的所有司法管辖区

## 3. 定义

**"儿童"** 是指任何18岁以下，或相关司法管辖区法定成年年龄以下（以较高者为准）的个人。

**"数字同意年龄"** 是指儿童可以独立同意处理其个人数据以用于数字服务的最低年龄，由适用司法管辖区规定。

**"父母/监护人"** 是指父母、法定监护人或其他依法有权代表儿童行事的人员。

**"第4级验证"** 是指协议的专业验证（成人+儿童）级别，其中持牌专业验证人员在父母/监护人的参与和同意下确认儿童的身份。

**"年龄范围证明"** 是指使用Bulletproofs的零知识证明，用以证明用户的年龄落在指定范围内，而无需披露确切出生日期。

**"Signet IQ"** 是指协议的连续身份识别商数评分（0–200），反映用户验证信号的累积强度。对于儿童，Signet IQ根据监护人自身的验证级别和Signet IQ、儿童自身的第4级凭证以及来自已验证账户的同伴证明进行计算。

## 5. 通过零知识证明进行年龄验证

### 5.1 年龄验证的工作原理

Signet协议使用基于Bulletproofs的零知识证明进行年龄验证。该方法：

1. **在不披露的情况下证明：** 用户可以证明其年龄落在指定范围内（例如，"13岁以上"、"16岁以上"、"18岁以上"、"13–17岁"），而无需披露确切出生日期。
2. **无法被逆向工程：** 零知识证明在数学上的构造使得无法从证明中提取底层出生日期。
3. **经过密码学验证：** 依赖方可以在不获取任何超出年龄范围声明之外信息的情况下验证证明的有效性。
4. **一次签发，多次使用：** 单个年龄范围证明可以向多个依赖方出示，无需重新验证。

### 5.2 年龄范围类别

协议支持以下标准年龄范围证明：

| 范围 | 描述 | 使用场景 |
|-------|-------------|----------|
| 13岁以下 | 低于COPPA门槛 | 需要完整的父母同意和监督 |
| 13–15岁 | 高于COPPA，低于部分GDPR门槛 | 根据司法管辖区可能需要父母同意 |
| 16–17岁 | 高于大多数数字同意年龄，低于成年年龄 | 大多数司法管辖区可独立同意；部分需要父母监督 |
| 18岁以上 | 大多数司法管辖区的成年年龄 | 完全独立同意 |
| 自定义 | 特定司法管辖区的年龄范围 | 符合特定地方要求 |

### 5.3 儿童证明签发

对于儿童（18岁以下），验证仪式遵循以用户为主导、专业人员确认的流程：

1. 父母/监护人和儿童与持牌专业人员（律师、公证员、医生或同等专业人员）进行第4级验证。
2. **用户（父母/监护人）在My Signet应用中输入所有文件详情：** 文件类型、签发国家、文件编号和儿童出生日期。验证人员检查实体文件以确认输入数据的准确性——他们不代表用户输入数据。
3. 协议使用`signet-nullifier-v2`公式（对长度前缀的docType、country、docNumber和"signet-nullifier-v2"进行SHA-256哈希）在用户设备本地计算基于文件的nullifier。nullifier被传输给验证人员；原始文件详情不会被传输。
4. 验证人员生成Bulletproof年龄范围证明，并同时签发两个凭证：自然人凭证（密钥对A）和人格凭证（密钥对B）。两者均携带年龄范围证明和监护人标签（`["guardian", "<parent_pubkey>"]`）。儿童的真实姓名仅作为私有Merkle叶子存储——从不公开发布。
5. 儿童自然人凭证的Merkle树包含：出生日期（用于年龄范围证明）、监护关系、文件类型、文件编号和文件有效期。儿童的姓名作为私有Merkle叶子包含在内，**不会**发布到任何中继。国籍可以包含在内以符合司法管辖区合规要求。
6. 基于文件的nullifier（使用`signet-nullifier-v2`对文件类型、国家和编号进行哈希）仅包含在自然人凭证中，防止重复身份创建。
7. **生物特征账户保护：** 在儿童的设备上，My Signet应用使用WebAuthn生物特征解锁（面部识别或指纹）来保护儿童账户的访问。生物特征数据不会离开设备——它完全由设备的安全飞地通过WebAuthn标准处理。不会向任何服务器传输或在设备外存储生物特征模板。

### 5.4 教师作为儿童验证的主要渠道

教师和学校管理人员是第4级儿童验证的公认渠道。学校入学记录——建立了亲子关系并提供出生日期证明——被接受作为父母身份证明的支持文件。

持牌专业人员教师（例如，在将教学视为受监管职业的司法管辖区中具有资质的教师，或学校关联的公证员）可以使用以下材料进行第4级验证：
- 父母/监护人的政府颁发身份证件
- 学校入学记录作为儿童证明（作为出生证明或护照的补充或替代）

这一途径对于家庭教育社区和律师或公证员难以获取的地区尤为重要。教育工作者的专业机构及其任职机构是信任锚。教师的虚假证明构成适用教育法下的职业不当行为。

### 5.5 面向网站的基于SDK的年龄验证

Signet SDK允许网站和应用程序在不接收任何个人身份信息的情况下验证儿童的年龄范围。数据流程为：

```
网站 → SDK → My Signet应用 → 零知识年龄范围证明 → 网站
```

网站请求特定年龄声明的证明（例如，"用户未满18岁"或"用户在13至17岁之间"）。Signet SDK将此请求路由到用户设备上的My Signet应用。应用程序在本地生成证明，仅返回密码学证明——没有姓名、没有出生日期、没有文件详情、没有nullifier。网站以数学方式验证证明，仅获知年龄范围声明。

这意味着网站可以遵守年龄验证要求（COPPA、英国网络安全法、澳大利亚16岁以下法规），而无需收集或存储儿童的任何个人数据。

## 6. 父母同意机制

### 6.1 按司法管辖区的同意要求

| 司法管辖区 | 数字同意年龄 | 所需同意机制 |
|-------------|--------------------|-----------------------------|
| 美国（COPPA） | 13岁 | 可验证的父母同意（例如，签署表格、信用卡验证、视频通话、政府身份证核查） |
| 欧盟（GDPR默认） | 16岁 | 合理努力核实父母同意 |
| 欧盟（成员国最低要求） | 13岁 | 按成员国实施规定 |
| 英国 | 13岁 | 适龄核实父母同意 |
| 澳大利亚 | 无特定数字年龄 | 基于成熟度和风险的合理步骤 |
| 韩国 | 14岁 | 法定代理人同意 |
| 巴西 | 12岁（父母同意至18岁） | 父母/监护人的专门和突出同意 |
| 印度 | 18岁 | 父母/监护人的可验证同意 |
| 中国 | 14岁 | 父母/监护人的单独同意 |
| 日本 | 15岁（指南） | 建议父母参与 |

### 6.2 协议同意机制

协议支持以下父母同意机制：

1. **第3/4级已验证父母凭证：**
   父母/监护人通过两凭证仪式（自然人+人格）获得第3级或第4级凭证以验证自身身份。然后，他们以密码学方式签署授权儿童凭证的同意事件。这创建了一个可验证的链：
   - 父母身份由专业验证人员验证（两凭证仪式）
   - 父母同意以其Nostr密钥进行密码学签名
   - 儿童凭证通过自然人和人格凭证上的不可变监护人标签与父母关联
   - 监护委托事件（kind 31000）允许父母将特定权限委托给其他受信任的成年人

2. **联合验证：**
   父母/监护人和儿童共同参加第4级验证会话。专业验证人员：
   - 验证父母/监护人和儿童的身份
   - 确认亲子或监护关系
   - 将父母/监护人的同意作为验证事件的一部分进行记录

3. **委托同意：**
   对于需要特定同意机制的司法管辖区（例如，COPPA的可验证父母同意），可能需要额外的验证步骤：
   - 签署的同意表格（数字或实体）
   - 视频验证会话
   - 同意父母/监护人的政府颁发身份证核查

### 6.3 同意撤回

父母/监护人可随时撤回同意，方式如下：
1. 从父母/监护人的Nostr密钥发布撤销事件（kind 31000），撤销同意事件。
2. 联系Signet协议支持团队请求撤销协助。
3. 通过实施协议同意管理功能的任何Nostr客户端。

撤回同意后：
- 儿童的凭证立即被撤销。
- 依赖方通过撤销事件收到通知。
- 与儿童相关的所有集中持有数据将在30天内删除，法律要求保留的情况除外。

## 7. 儿童数据最小化

### 7.1 原则

协议对儿童数据应用最严格的数据最小化原则。收集或处理的数据不超过凭证验证所严格必要的范围。

### 7.2 收集内容

| 数据 | 是否收集 | 理由 |
|------|-----------|-----------|
| 儿童的Nostr公钥 | 是 | 凭证绑定所必需 |
| 年龄范围证明（零知识证明） | 是 | 年龄验证所必需 |
| 父母同意记录 | 是 | 法律合规所必需 |
| 凭证元数据（级别、日期） | 是 | 凭证功能所必需 |
| 儿童姓名 | **否（不上链）** | 仅作为私有Merkle叶子存储；从不发布 |
| 儿童出生日期 | **否（不上链）** | 由零知识年龄范围证明替代；仅作为私有Merkle叶子存储 |
| 儿童照片 | **否** | 不需要 |
| 儿童政府身份证号码 | **否（不上链）** | 仅作为私有Merkle叶子存储以供选择性披露；验证仪式后不由验证方存储 |
| Nullifier哈希（`signet-nullifier-v2`） | 是（仅自然人凭证） | 防止重复所必需；无法逆向还原为文件详情 |
| 监护人公钥 | 是（两个凭证） | 监护人与儿童关联所必需 |
| 监护关系 | 是（私有Merkle叶子，自然人凭证） | 建立监护人关联的法律依据 |
| Merkle根 | 是（仅自然人凭证） | 提交已验证属性而不发布它们 |
| 实体类型标签 | 是（两个凭证） | 区分自然人和人格所必需 |
| 儿童位置 | **否** | 不需要 |
| 儿童学校或机构 | **否（不上链）** | 可作为支持证据参考，但不发布 |
| 行为数据 | **否** | 禁止 |
| 使用追踪 | **否** | 儿童禁止 |
| 生物特征数据 | **否** | 仅通过WebAuthn在设备本地处理；不传输模板 |

### 7.3 儿童的Merkle树属性

儿童自然人凭证提交到具有以下叶子的Merkle树：

| 叶子 | 目的 | 是否上链披露？ |
|------|---------|---------------------|
| `dateOfBirth:<ISO日期>` | 零知识年龄范围证明的来源 | 否——私有 |
| `guardianRelationship:<类型>` | 法律依据（例如，"parent"、"legal_guardian"） | 否——私有 |
| `documentType:<类型>` | 提交证明的类型 | 否——私有 |
| `documentNumber:<编号>` | 如需要可进行选择性披露 | 否——私有 |
| `documentExpiry:<日期>` | 文件有效性检查 | 否——私有 |
| `name:<姓名>` | 如需要可进行选择性披露 | 否——私有 |
| `nationality:<代码>` | 司法管辖区合规 | 否——私有 |
| `nullifier:<哈希>` | 防止重复 | 是（作为顶层标签） |

只有Merkle根和nullifier哈希出现在链上。只有当儿童的监护人明确行使选择性披露权（例如，向监管机构证明儿童的国籍）时，才会披露各个叶子。

### 7.4 儿童的Signet IQ

儿童的Signet IQ评分由以下内容计算：
- 第4级凭证本身（主要贡献）
- 监护人自身的验证级别和Signet IQ（加权贡献，反映监护关系的强度）
- 认识该家庭的其他第2级以上账户的同伴证明

儿童的Signet IQ作为综合信任信号向依赖方显示。它不会揭示监护人的身份、儿童的姓名或任何其他个人属性——它仅从公开凭证事件中计算。禁止基于儿童的Signet IQ在年龄验证和访问控制以外的目的对儿童进行社会评分（见第8节）。

### 7.5 保留限制

儿童数据须遵守最短允许保留期：
- 有效凭证：直至到期、撤销或儿童达到成年年龄（以最早者为准）
- 同意记录：凭证有效期加任何法律要求的保留期
- 验证记录：按验证方专业义务要求（通常为1–3年）
- 挑战/响应数据：验证后立即删除

## 8. 禁止使用

### 8.1 绝对禁止

以下与儿童相关的协议使用行为被**严格禁止**：

1. **建立档案：** 将协议用于出于任何目的（包括营销、广告或行为分析）对儿童建立档案。
2. **追踪：** 使用凭证或验证数据追踪儿童的在线或线下活动。
3. **定向广告：** 使用年龄范围证明或凭证数据向儿童投放定向广告。
4. **数据货币化：** 出售、许可或以其他方式将儿童的凭证数据或年龄验证数据货币化。
5. **自动化决策：** 使用儿童的凭证数据进行产生法律或类似重大影响的自动化决策。
6. **监控：** 使用协议对儿童活动进行持续监控。
7. **社会评分：** 使用凭证或Signet IQ数据在适龄访问控制以外对儿童创建社会评分系统。
8. **暗示：** 使用利用儿童弱点或操纵其行为的设计模式。
9. **不必要的数据收集：** 从儿童收集超过凭证验证所严格必要的数据。
10. **未经同意分享：** 在没有可验证的父母同意的情况下与第三方分享儿童数据。

### 8.2 依赖方限制

接收儿童年龄范围证明或凭证验证的依赖方：
- 不得将数据用于寻求验证的特定目的以外
- 不得在即时需要之外保留验证数据
- 必须实施适合其服务的儿童安全措施
- 必须遵守其司法管辖区所有适用的儿童保护法律

## 9. 事件报告和响应

### 9.1 可报告事件类型

以下涉及儿童的事件必须报告：

| 事件类型 | 报告门槛 | 优先级 |
|--------------|---------------------|----------|
| 儿童凭证数据泄露 | 任何未经授权的访问 | 严重——立即 |
| 儿童虚假第4级凭证 | 发现任何实例 | 严重——立即 |
| 父母同意机制被破坏 | 任何被破坏迹象 | 严重——立即 |
| 年龄验证被利用 | 发现任何绕过行为 | 高——4小时内 |
| 依赖方滥用儿童凭证 | 任何报告的滥用 | 高——4小时内 |
| 协议促成的疑似儿童剥削 | 任何迹象 | 严重——立即+执法机构 |
| 儿童身份去匿名化 | 任何实例 | 严重——立即 |

### 9.2 报告程序

**内部报告：**
- 所有事件立即报告给指定儿童安全官员。
- 儿童安全官员向数据保护官员（DPO）和高级管理层上报。

**监管报告：**
- 在适用法律规定的时限内通知相关监督机构（例如，GDPR下72小时）。
- 遵循特定国家的通知要求（见第10节）。

**执法报告：**
- 涉及疑似儿童剥削的事件立即报告给：
  - 当地执法机构
  - 美国国家失踪与受剥削儿童中心（NCMEC）（美国）
  - 互联网监察基金会（IWF）（英国）
  - 相关司法管辖区的同等机构

**父母通知：**
- 在不必要延迟的情况下将任何影响其子女数据或凭证的事件通知父母/监护人。
- 通知内容包括事件描述、潜在影响和正在采取的措施。

### 9.3 修复

儿童安全事件后：
1. 立即控制事件
2. 撤销受影响的凭证
3. 法证调查
4. 根本原因分析
5. 实施预防措施
6. 对受影响方进行后续通知
7. 事后审查，如需要则更新政策

### 10.1 英国

| 标准 | 实施 |

| 数据保护影响评估 | 对第4级及所有与儿童相关的处理进行DPIA |
| 适龄应用 | 为不同年龄组设计协议界面 |
| 透明度 | 以适龄语言提供隐私信息 |

| 政策和社区标准 | 明确儿童凭证使用方式的标准 |

### 10.2 美国

**COPPA合规：**
- 在从13岁以下儿童收集任何信息之前获得可验证的父母同意

- 父母可访问、审查和删除儿童数据
- 不得以不必要的信息披露为条件允许儿童参与
- 对儿童数据采取合理的安全措施

**FTC 2026年3月立场：** 对于仅出于年龄验证目的收集个人数据的情况，如果数据被可靠删除并给予明确通知，FTC将不采取执法行动。Signet超越这一标准：在年龄验证期间不收集、存储或传输任何个人数据——零知识证明不包含任何个人身份信息。

**州法律：**

- 其他适用的州法律

**报告：** 向FTC和适用的州检察长报告。

## 11. 儿童安全设计原则

### 11.1 隐私设计

所有影响儿童的协议功能均包含：
- 架构层面的数据最小化
- 零知识证明以避免暴露个人数据
- 内置于凭证结构中的密码学保护
- 默认隐私保护设置
- 用户输入-验证方确认的仪式流程——文件数据由用户在其自己的设备上输入；验证方确认准确性但从不持有原始文件详情

### 11.2 安全设计

协议界面和实施应：
- 明确区分儿童和成人凭证工作流程
- 防止意外暴露儿童身份信息
- 包含防止凭证滥用的保护措施
- 提供清晰的适龄指导
- 使用生物特征设备解锁（WebAuthn）保护设备上的儿童账户

### 11.3 定期审查

本政策和所有儿童安全措施的审查时机：
- 至少每年一次
- 适用法律发生任何变化时
- 任何儿童安全事件发生后
- 协议发生重大变化时

### 11.4 监护委托模型

协议实施三层家庭结构模型：

**第1层——凭证级别（不可变）：**
监护人标签（`["guardian", "<parent_pubkey>"]`）由专业验证人员设置，反映法定父母责任。只能通过由具有适当法律文件（例如，法院命令）的专业人员签发新凭证来更改。

**第2层——委托级别（灵活）：**
监护人可以通过kind 31000监护委托事件将特定权限委托给受信任的成年人（继父母、祖父母、教师）。委托具有：
- 时间限制（有到期日）
- 范围限制：`full`、`activity-approval`、`content-management`、`contact-approval`
- 监护人可随时撤销
- 由监护人的Nostr密钥签名
- 标记为`["agent-type", "guardian"]`以区别于其他委托事件类型

**第3层——客户端级别（特定应用）：**
应用程序根据第1层和第2层数据执行权限，包括屏幕时间限制、内容过滤、活动审批工作流程和联系人限制。

## 12. 培训和意识

### 12.1 验证方培训

所有授权执行第4级（儿童）验证的专业验证人员必须：
- 在执行儿童验证之前完成儿童安全培训
- 了解其司法管辖区的儿童保护法律
- 了解保障指标和报告义务
- 了解用户输入-验证方确认的仪式流程，不尝试代表用户输入数据
- 每年刷新培训

### 12.2 人员培训

参与协议开发、运营或支持的所有人员必须：
- 完成儿童安全意识培训
- 了解本政策及其义务
- 知道如何报告儿童安全问题
